6. **Understanding _version Metadata** (FOUN-07) -- This is a CRITICAL section that iOS/Android guides omit entirely. Cover:

   - **Why these fields exist:** DataStore enables conflict resolution on the AppSync backend. This adds three metadata fields to every model: `_version` (optimistic locking counter), `_deleted` (soft-delete flag), `_lastChangedAt` (timestamp).
   - **When you need them:** If your app used DataStore, your backend HAS conflict resolution enabled. All mutations REQUIRE `_version` in the input. Omitting it causes `ConditionalCheckFailedException`.
   - **How to handle them:**
     - Always include `_version`, `_deleted`, `_lastChangedAt` in query/mutation response selections
     - Always pass `_version` from the last query result into mutation inputs
     - Filter soft-deleted records: `items.filter(item => !item._deleted)`
   - **Example: What happens without _version:**
```
// WITHOUT _version in mutation input:
// Error: "ConditionalCheckFailedException" or "ConflictUnhandled"

// WITH _version:
// Success: mutation completes, returns new _version value
```
   - **Helper pattern** -- A simple utility function:
```typescript
// Filter out soft-deleted records from list query results
function filterDeleted<T extends { _deleted?: boolean | null }>(items: T[]): T[] {
  return items.filter(item => !item._deleted);
}
```
   - **Can I disable conflict resolution?** -- Brief note: yes, but it requires backend changes (modifying the data resource and redeploying). This guide assumes conflict resolution is still active. Link to Amplify docs for disabling it.

Write `guide/04-apollo-setup.md` with these sections:

1. **Title** -- `# Apollo Client Setup`

2. **Overview** -- Brief intro: this page configures Apollo Client to work with your AppSync endpoint using Cognito User Pools authentication. The setup includes auth token injection, error handling, retry logic, and sign-out cleanup.

3. **The HTTP Link** (FOUN-01 partial) -- Create the HTTP connection to AppSync:
```typescript
import { createHttpLink } from '@apollo/client';
import outputs from '../amplify_outputs.json';

const httpLink = createHttpLink({
  uri: outputs.data.url,
});
```
Explain: `outputs.data.url` is the GraphQL endpoint from `amplify_outputs.json`. Do NOT use `BatchHttpLink` -- AppSync does not support HTTP request batching.

4. **The Auth Link** (FOUN-02) -- Cognito token injection via `setContext`:
```typescript
import { setContext } from '@apollo/client/link/context';
import { fetchAuthSession } from 'aws-amplify/auth';

const authLink = setContext(async (_, { headers }) => {
  try {
    const session = await fetchAuthSession();
    const token = session.tokens?.idToken?.toString();
    return {
      headers: {
        ...headers,
        authorization: token || '',
      },
    };
  } catch (error) {
    console.error('Auth session error:', error);
    return { headers };
  }
});
```
Explain:
- `fetchAuthSession()` is called per-request, ensuring fresh tokens automatically
- The try/catch handles token expiry gracefully -- if the refresh token is also expired, the request proceeds without auth (AppSync will return an auth error, which the error link handles)
- Uses `idToken` (not `accessToken`) because AppSync Cognito User Pools auth expects the ID token in the `authorization` header

5. **The Error Link** (FOUN-05) -- Global error handling:
```typescript
import { onError } from '@apollo/client/link/error';

const errorLink = onError(({ graphQLErrors, networkError }) => {
  if (graphQLErrors) {
    for (const { message, locations, path } of graphQLErrors) {
      console.error(
        `[GraphQL error]: Message: ${message}, Location: ${locations}, Path: ${path}`
      );
      // Handle specific error types
      if (message.includes('Unauthorized') || message.includes('401')) {
        // Token expired or invalid -- redirect to sign-in
        // window.location.href = '/signin';
      }
    }
  }
  if (networkError) {
    console.error(`[Network error]: ${networkError}`);
  }
});
```
Explain: The error link intercepts ALL GraphQL and network errors before they reach components. Show how to detect auth errors and redirect to sign-in.

6. **The Retry Link** (FOUN-05) -- Automatic retry with backoff:
```typescript
import { RetryLink } from '@apollo/client/link/retry';

const retryLink = new RetryLink({
  delay: {
    initial: 300,
    max: 5000,
    jitter: true,
  },
  attempts: {
    max: 3,
    retryIf: (error) => !!error,
  },
});
```
Explain: Retries failed network requests up to 3 times with jittered exponential backoff (300ms, ~600ms, ~1200ms). The `jitter: true` prevents thundering herd.

7. **Putting It All Together** (FOUN-01 complete) -- The complete Apollo Client configuration:
```typescript
import {
  ApolloClient,
  InMemoryCache,
  createHttpLink,
  from,
} from '@apollo/client';

export const apolloClient = new ApolloClient({
  link: from([retryLink, errorLink, authLink, httpLink]),
  cache: new InMemoryCache(),
});
```
Explain the link chain order: `from()` executes left to right on requests. RetryLink wraps the chain (retries on failure), errorLink intercepts errors, authLink injects the token, httpLink sends to AppSync. Show a diagram of the link chain:
```
Request → RetryLink → ErrorLink → AuthLink → HttpLink → AppSync
Response ← RetryLink ← ErrorLink ← AuthLink ← HttpLink ← AppSync
```

async function handleSignOut() {
