1. **Title** -- `# Real-Time Subscriptions`

2. **Why Not Apollo Subscriptions?** -- Explain the core architectural decision:
   - AppSync uses a custom WebSocket subprotocol (not standard `graphql-ws` or `subscriptions-transport-ws`)
   - On iOS/Android, the AWS AppSync Apollo Extensions library handles this protocol. **No JavaScript equivalent exists.**
   - Standard Apollo subscription libraries (`graphql-ws`, `subscriptions-transport-ws`) will fail silently -- WebSocket connects then immediately disconnects with no helpful error message
   - Amplify Gen 2 already has a production-tested implementation (`AWSAppSyncRealTimeProvider`) that handles auth, reconnection, and token refresh
   - **Solution:** Use Amplify for subscriptions, Apollo for everything else

Include a warning callout:
```markdown
> **Warning:** Do NOT use `graphql-ws`, `subscriptions-transport-ws`, or Apollo's `WebSocketLink` with AppSync.
> These libraries do not speak AppSync's custom WebSocket protocol and will fail silently.
```

5. **Pattern 1: Refetch on Subscription Event (Recommended)** -- The primary pattern per locked decision. Show a complete React component:
```typescript
import { useQuery } from '@apollo/client';
import { generateClient } from 'aws-amplify/api';
import { useEffect } from 'react';
import { LIST_POSTS } from './graphql/queries';

const amplifyClient = generateClient();

function PostList() {
  const { data, loading, error, refetch } = useQuery(LIST_POSTS);

  useEffect(() => {
    const subscriptions = [
      amplifyClient.graphql({
        query: `subscription OnCreatePost {
          onCreatePost { id }
        }`
      }).subscribe({
        next: () => refetch(),
        error: (err) => console.error('Create subscription error:', err),
      }),
      amplifyClient.graphql({
        query: `subscription OnUpdatePost {
          onUpdatePost { id }
        }`
      }).subscribe({
        next: () => refetch(),
        error: (err) => console.error('Update subscription error:', err),
      }),
      amplifyClient.graphql({
        query: `subscription OnDeletePost {
          onDeletePost { id }
        }`
      }).subscribe({
        next: () => refetch(),
        error: (err) => console.error('Delete subscription error:', err),
      }),
    ];

    return () => subscriptions.forEach(sub => sub.unsubscribe());
  }, [refetch]);

  // ... render logic
}
```
Explain:
- The subscription payload only needs `id` since we refetch the full list anyway
- This pattern is simple and reliable but causes an extra network round-trip per event
- For most apps, this latency is imperceptible

6. **Pattern 2: Direct Cache Update (Advanced)** -- Brief section showing how to update Apollo's cache directly from subscription data instead of refetching. This is more performant but more complex:
```typescript
useEffect(() => {
  const sub = amplifyClient.graphql({
    query: `subscription OnCreatePost {
      onCreatePost { id title content _version _deleted _lastChangedAt createdAt updatedAt }
    }`
  }).subscribe({
    next: ({ data }) => {
      const newPost = data.onCreatePost;
      // Update Apollo cache directly
      apolloClient.cache.modify({
        fields: {
          listPosts(existingData = { items: [] }) {
            const newRef = apolloClient.cache.writeFragment({
              data: newPost,
              fragment: POST_DETAILS_FRAGMENT,
            });
            return {
              ...existingData,
              items: [...existingData.items, newRef],
            };
          },
        },
      });
    },
  });
  return () => sub.unsubscribe();
}, []);
```
Note: This pattern requires the full payload (not just `id`). Mark as advanced and recommend Pattern 1 for most use cases.

9. **Troubleshooting** -- Common issues:
   - "Subscription connects but never fires": Check that the subscription name matches the schema (e.g., `onCreatePost` not `onPostCreated`)
   - "Auth error on subscription": Ensure Amplify is configured before creating the client. `Amplify.configure(outputs)` must run before `generateClient()`.
   - "Subscription disconnects after ~5 minutes of inactivity": This is normal -- Amplify's provider handles automatic reconnection

4. **During Migration Checklist** -- Steps to follow while migrating each feature. Organized by operation type:

```markdown
<!-- ai:checklist:during-migration -->
### During Migration Checklist

**For each DataStore model:**

- [ ] **Define GraphQL fragment** -- Create a `ModelDetails` fragment including all fields + `_version`, `_deleted`, `_lastChangedAt`
- [ ] **Define GraphQL operations** -- Create list, get, create, update, delete operations using the fragment
- [ ] **Migrate queries** -- Replace `DataStore.query(Model)` with `useQuery(LIST_MODEL)` or `apolloClient.query()`
- [ ] **Migrate single-item queries** -- Replace `DataStore.query(Model, id)` with `useQuery(GET_MODEL, { variables: { id } })`
- [ ] **Migrate creates** -- Replace `DataStore.save(new Model({...}))` with `useMutation(CREATE_MODEL)` or `apolloClient.mutate()`
- [ ] **Migrate updates** -- Replace `DataStore.save(Model.copyOf(...))` with update mutation. **Include `_version` in input.**
- [ ] **Migrate deletes** -- Replace `DataStore.delete(instance)` with delete mutation. **Include `id` and `_version` in input.**
- [ ] **Filter soft-deleted records** -- Add `_deleted` filter to list queries or use the `filterDeleted` helper
- [ ] **Migrate observe** -- Replace `DataStore.observe(Model)` with Amplify subscription + refetch. See [subscriptions](./05-subscriptions.md)
- [ ] **Migrate observeQuery** -- Replace `DataStore.observeQuery(Model)` with `useQuery` + subscription refetch
- [ ] **Update error handling** -- Replace DataStore error patterns with Apollo error link and component-level error handling
- [ ] **Test each migrated operation** -- Verify create, read, update, delete, and real-time all work

**For predicates and filters:**
- [ ] **Migrate filter syntax** -- Convert DataStore predicates (`c => c.status.eq('ACTIVE')`) to GraphQL filter objects (`{ filter: { status: { eq: 'ACTIVE' } } }`)
- [ ] **Migrate sorting** -- Convert DataStore sort to GraphQL `sortDirection` and `sortField` parameters
- [ ] **Migrate pagination** -- Convert DataStore `.page()` to GraphQL `nextToken` / `limit` parameters
```
