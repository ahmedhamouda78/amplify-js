3. **Component Migration: Imperative to Declarative** (`<!-- ai:component-migration -->`) -- REAC-01
   - Show a COMPLETE before/after component migration. This is the centerpiece example.
   - **Before:** A typical DataStore React component:
     ```typescript
     function PostList() {
       const [posts, setPosts] = useState<Post[]>([]);
       const [loading, setLoading] = useState(true);

       useEffect(() => {
         setLoading(true);
         DataStore.query(Post).then(results => {
           setPosts(results);
           setLoading(false);
         });
       }, []);

       const handleDelete = async (post: Post) => {
         await DataStore.delete(post);
         setPosts(prev => prev.filter(p => p.id !== post.id));
       };

       if (loading) return <p>Loading...</p>;
       return (
         <ul>
           {posts.map(post => (
             <li key={post.id}>
               {post.title}
               <button onClick={() => handleDelete(post)}>Delete</button>
             </li>
           ))}
         </ul>
       );
     }
     ```
   - **After:** The same component using Apollo hooks:
     ```typescript
     function PostList() {
       const { data, loading, error } = useQuery(LIST_POSTS);
       const [deletePost] = useMutation(DELETE_POST, {
         refetchQueries: [{ query: LIST_POSTS }],
       });

       const handleDelete = async (post: any) => {
         await deletePost({
           variables: { input: { id: post.id, _version: post._version } },
         });
       };

       if (loading) return <p>Loading...</p>;
       if (error) return <p>Error: {error.message}</p>;

       const posts = data?.listPosts?.items?.filter((p: any) => !p._deleted) || [];
       return (
         <ul>
           {posts.map((post: any) => (
             <li key={post.id}>
               {post.title}
               <button onClick={() => handleDelete(post)}>Delete</button>
             </li>
           ))}
         </ul>
       );
     }
     ```
   - Explain key differences:
     - No `useState` for data/loading -- Apollo hooks manage this
     - No `useEffect` for initial fetch -- `useQuery` fetches on mount
     - No manual state updates after mutations -- `refetchQueries` handles it
     - Error state is now available (DataStore swallowed errors)
     - Must filter `_deleted` records

4. **Loading and Error States** (`<!-- ai:loading-error -->`) -- REAC-02
   - Explain: DataStore had NO loading state concept. `DataStore.query()` returned a Promise, but there was no built-in way to know if initial sync was complete. Apollo hooks provide explicit `loading`, `error`, and `data` states.
   - Show the three states from `useQuery`:
     ```typescript
     const { data, loading, error } = useQuery(LIST_POSTS);
     // loading: true while fetch is in-flight
     // error: ApolloError if query failed
     // data: the query result (undefined until first successful fetch)
     ```
   - Show pattern for `cache-and-network` fetchPolicy where both loading and data can be truthy:
     ```typescript
     const { data, loading, error } = useQuery(LIST_POSTS, {
       fetchPolicy: 'cache-and-network',
     });
     // On first load: loading=true, data=undefined
     // On cache hit + refetch: loading=true, data=cached (show cached while refreshing)
     if (loading && !data) return <p>Loading...</p>; // Only show spinner on first load
     ```
   - Show `useMutation` loading/error:
     ```typescript
     const [createPost, { loading: creating, error: createError }] = useMutation(CREATE_POST);
     // creating: true while mutation is in-flight
     // Disable submit button while creating
     <button disabled={creating}>{creating ? 'Creating...' : 'Create'}</button>
     ```
   - Show error boundary pattern for unrecoverable errors.

5. **Migrating DataStore.observe()** (`<!-- ai:observe-migration -->`) -- RTOB-01
   - Explain: DataStore.observe() returned a single Observable with `opType` (INSERT/UPDATE/DELETE). The migration uses three separate Amplify subscriptions, one for each event type.
   - Before:
     ```typescript
     useEffect(() => {
       const sub = DataStore.observe(Post).subscribe(msg => {
         if (msg.opType === 'INSERT') { /* handle create */ }
         if (msg.opType === 'UPDATE') { /* handle update */ }
         if (msg.opType === 'DELETE') { /* handle delete */ }
       });
       return () => sub.unsubscribe();
     }, []);
     ```
   - After (using refetch pattern from Phase 1 subscriptions page):
     ```typescript
     const amplifyClient = generateClient();

     function PostList() {
       const { data, loading, error, refetch } = useQuery(LIST_POSTS);

       useEffect(() => {
         const subscriptions = [
           amplifyClient.graphql({
             query: `subscription { onCreatePost { id } }`,
           }).subscribe({ next: () => refetch() }),
           amplifyClient.graphql({
             query: `subscription { onUpdatePost { id } }`,
           }).subscribe({ next: () => refetch() }),
           amplifyClient.graphql({
             query: `subscription { onDeletePost { id } }`,
           }).subscribe({ next: () => refetch() }),
         ];
         return () => subscriptions.forEach(sub => sub.unsubscribe());
       }, [refetch]);

       // ... render
     }
     ```
   - Cross-reference: "For advanced cache update patterns instead of refetch, see [Subscriptions - Pattern 2](./05-subscriptions.md#pattern-2-direct-cache-update-advanced)."
   - Note: DataStore's observe supported per-ID and per-predicate filtering. For per-ID observation, subscribe to all events and filter in the callback: `if (data.onUpdatePost.id === targetId) { refetch(); }`

6. **Migrating DataStore.observeQuery()** (`<!-- ai:observe-query-migration -->`) -- RTOB-02
   - Explain: `observeQuery()` combined an initial query with live updates into a single Observable that emitted snapshots. The Apollo equivalent is `useQuery` with `fetchPolicy: 'cache-and-network'` plus subscription-triggered refetch.
   - Before:
     ```typescript
     useEffect(() => {
       const sub = DataStore.observeQuery(Post, p => p.status.eq('PUBLISHED'), {
         sort: s => s.createdAt(SortDirection.DESCENDING),
       }).subscribe(snapshot => {
         setPosts(snapshot.items);
         setIsSynced(snapshot.isSynced);
       });
       return () => sub.unsubscribe();
     }, []);
     ```
   - After:
     ```typescript
     function PublishedPosts() {
       const { data, loading, refetch } = useQuery(LIST_POSTS, {
         variables: { filter: { status: { eq: 'PUBLISHED' } } },
         fetchPolicy: 'cache-and-network',
       });

       useEffect(() => {
         const subscriptions = [
           amplifyClient.graphql({
             query: `subscription { onCreatePost { id } }`,
           }).subscribe({ next: () => refetch() }),
           amplifyClient.graphql({
             query: `subscription { onUpdatePost { id } }`,
           }).subscribe({ next: () => refetch() }),
           amplifyClient.graphql({
             query: `subscription { onDeletePost { id } }`,
           }).subscribe({ next: () => refetch() }),
         ];
         return () => subscriptions.forEach(sub => sub.unsubscribe());
       }, [refetch]);

       const posts = data?.listPosts?.items
         ?.filter((p: any) => !p._deleted)
         ?.sort((a: any, b: any) =>
           new Date(b.createdAt).getTime() - new Date(a.createdAt).getTime()
         ) || [];

       return (/* render */);
     }
     ```
   - Key differences table:
     | observeQuery feature | Apollo equivalent | Notes |
     |---------------------|-------------------|-------|
     | Initial snapshot | `useQuery` first result | Same behavior |
     | Live updates | Subscription refetch | Each event triggers full refetch |
     | `isSynced` flag | No equivalent | API Only has no sync engine |
     | Debounced snapshots | No debouncing | Each subscription event triggers immediate refetch |
     | Predicate filtering | `filter` variable | Same filtering, different syntax |
     | Sort option | Client-side `.sort()` | See Predicates page |

7. **Owner-Based Auth Subscriptions** (`<!-- ai:owner-auth -->`) -- RTOB-03
   - Explain: When a model has `@auth(rules: [{ allow: owner }])`, DataStore automatically injected the `owner` argument into subscriptions. After migration, you MUST manually pass the `owner` variable or the subscription will silently receive no events.
   - Show how to get the owner value:
     ```typescript
     import { fetchAuthSession } from 'aws-amplify/auth';

     async function getCurrentOwner(): Promise<string> {
       const session = await fetchAuthSession();
       // Default Amplify Gen 2 uses 'sub' claim
       return session.tokens?.idToken?.payload?.sub as string;
     }
     ```
   - Show complete component with owner-based subscriptions:
     ```typescript
     function MyPosts() {
       const { data, refetch } = useQuery(LIST_POSTS);
       const [owner, setOwner] = useState<string | null>(null);

       useEffect(() => {
         getCurrentOwner().then(setOwner);
       }, []);

       useEffect(() => {
         if (!owner) return;
         const subscriptions = [
           amplifyClient.graphql({
             query: `subscription OnCreatePost($owner: String!) {
               onCreatePost(owner: $owner) { id }
             }`,
             variables: { owner },
           }).subscribe({ next: () => refetch() }),
           // ... onUpdatePost and onDeletePost with same owner variable
         ];
         return () => subscriptions.forEach(sub => sub.unsubscribe());
       }, [owner, refetch]);
       // ... render
     }
     ```
   - Show the full subscription for all three event types (not abbreviated).
   - Warning callout: "If your subscriptions connect successfully but never fire events, check whether your model has owner-based authorization. This is the most common cause of silent subscription failures after migration."
   - Note: The owner claim may be `sub` (default) or `username` depending on Cognito configuration. Show how to check: look at the `ownerField` in the model's auth rules.
