   - After: `{ or: [{ status: { eq: 'PUBLISHED' } }, { status: { eq: 'DRAFT' } }] }`
   - Before: `p.status.notIn(['ARCHIVED', 'DELETED'])`
   - After: `{ and: [{ status: { ne: 'ARCHIVED' } }, { status: { ne: 'DELETED' } }] }`

4. **Logical Predicates (and, or, not)** (`<!-- ai:logical-predicates -->`) -- CRUD-07
   - Mapping table: DataStore `p.and(p => [...])` -> GraphQL `{ and: [{...}, {...}] }`
   - Before/after for `and`:
     - DataStore: `p.and(p => [p.rating.gt(4), p.status.eq('PUBLISHED')])`
     - GraphQL: `{ and: [{ rating: { gt: 4 } }, { status: { eq: 'PUBLISHED' } }] }`
   - Before/after for `or`:
     - DataStore: `p.or(p => [p.title.contains('React'), p.title.contains('Apollo')])`
     - GraphQL: `{ or: [{ title: { contains: 'React' } }, { title: { contains: 'Apollo' } }] }`
   - Before/after for `not`:
     - DataStore: `p.not(p => p.status.eq('DRAFT'))`
     - GraphQL: `{ not: { status: { eq: 'DRAFT' } } }`
   - Complex nested example combining and + or (use the exact example from research):
     ```
     // DataStore: p.and(p => [p.rating.gt(4), p.status.eq('PUBLISHED'), p.or(p => [p.title.contains('React'), p.title.contains('Apollo')])])
     // GraphQL: { and: [{ rating: { gt: 4 } }, { status: { eq: 'PUBLISHED' } }, { or: [{ title: { contains: 'React' } }, { title: { contains: 'Apollo' } }] }] }
     ```
   - Note: Top-level filter fields are implicitly AND-ed in AppSync (so `{ status: { eq: 'PUBLISHED' }, rating: { gt: 4 } }` is equivalent to using explicit `and`).

5. **Pagination Migration** (`<!-- ai:pagination -->`) -- CRUD-08
   - **Conceptual explanation first:** DataStore uses page-based (zero-indexed `page` + `limit`), AppSync uses cursor-based (`nextToken` + `limit`). This is NOT a rename -- it's a fundamental semantic change.
   - Key difference: Page-based allows random access (jump to page 5). Cursor-based is sequential only.
   - Before: `DataStore.query(Post, Predicates.ALL, { page: 0, limit: 10 })` then `{ page: 1, limit: 10 }`
   - After: `apolloClient.query({ query: LIST_POSTS, variables: { limit: 10 } })` then use `nextToken` from response for next page
   - Show complete "Load More" pattern in React:
     ```typescript
     function PostList() {
       const { data, loading, fetchMore } = useQuery(LIST_POSTS, { variables: { limit: 10 } });
       const handleLoadMore = () => {
         fetchMore({ variables: { limit: 10, nextToken: data.listPosts.nextToken } });
       };
       // ... render with "Load More" button disabled when nextToken is null
     }
     ```
   - Warning: `nextToken` with filters may return fewer items than `limit`. Always check `nextToken !== null` for more pages, not `items.length < limit`.
   - Note: For "jump to page" UX, must either fetch pages sequentially or redesign as infinite scroll.

6. **Sorting Migration** (`<!-- ai:sorting -->`) -- CRUD-09
   - Explain: DataStore supports `SortDirection.ASCENDING`/`DESCENDING` with chainable sort predicates. AppSync's basic `listModels` has NO `sortDirection` argument by default.
   - Two approaches:
     a. **Client-side sorting (recommended for most cases):** Fetch all results, sort in JavaScript.
        - Before: `DataStore.query(Post, Predicates.ALL, { sort: s => s.createdAt(SortDirection.DESCENDING) })`
        - After: `const posts = [...data.listPosts.items].filter(p => !p._deleted).sort((a, b) => new Date(b.createdAt).getTime() - new Date(a.createdAt).getTime())`
     b. **Server-side sorting (requires @index directive):** If the model has a `@index` directive on a field, AppSync generates a `listByField` query with `sortDirection` argument. Briefly mention this but note it requires backend schema changes.
   - Show multi-field sort example (sort by status then by createdAt).
